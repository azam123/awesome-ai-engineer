01 · Foundations

Generative AI vs. Agentic AI

Both are built on the same LLMs — the difference is what happens around the model: a single forward pass that produces content, versus a loop that pursues a goal.

Definition

Generative AI

A model that takes a prompt and produces new content — text, code, images, audio — in a single request/response cycle. It reacts; it does not pursue a goal beyond completing the current output.

Single-shotStateless by defaultNo tool use
Typical uses
  • Drafting an email or a blog post
  • Summarizing a document
  • Generating an image from a description
  • Autocompleting code in an editor
  • Answering a question from what the model already knows
Definition

Agentic AI

An LLM used as a reasoning engine inside a loop: it plans, chooses tools, observes results, and decides whether to continue or stop — pursuing a goal across multiple steps rather than answering once.

Multi-step loopTool / function callingMemory & stateSelf-correction
Typical uses
  • A coding agent that writes, runs, and fixes tests
  • A research agent that searches, reads, and synthesizes across sources
  • A support agent that checks order status, issues a refund, and confirms it
  • A DevOps agent that diagnoses an incident and opens a fix
  • An orchestrator that delegates to specialised sub-agents
DimensionGenerative AIAgentic AI
Interaction patternOne prompt → one responsePlan → act → observe → repeat
AutonomyNone — waits for the next promptPursues a goal until done or blocked
Tool useUsually noneCalls APIs, runs code, queries databases
MemoryContext window onlyWorking memory + long-term state across steps
PlanningImplicit, within one responseExplicit — decomposes tasks into sub-steps
Error handlingNone — output is finalCan observe failure and retry a different approach
Example architecturePrompt → LLM → outputReAct / plan-execute loop with tool calls
02 · RAG Core

What is RAG?

Retrieval Augmented Generation pairs an LLM with a retrieval step over your own data, so the model answers using facts it looks up at request time — not just what it memorized during training.

The problem it solves

LLMs are frozen at training time, can't see private or proprietary data, and will confidently hallucinate when they don't know something. Fine-tuning helps with style, not with fast-changing facts.

The core idea

Before generating an answer, retrieve the most relevant snippets of external knowledge and insert them into the prompt as context — the model then generates grounded on that evidence.

Why not just fine-tune?

Fine-tuning is expensive to keep current and doesn't reliably teach new facts. RAG keeps knowledge in an updatable index — refresh the data, not the model.

04 · Deep Dive

Step 1 — Chunking

Documents are too large to embed whole and too meaningful to slice blindly. Chunking finds the balance between context and precision. Try a strategy below.

01

Fixed-size

Split every N characters or tokens, usually with 10–20% overlap so context isn't cut at a boundary. Fast and predictable, but can slice a sentence in half.

02

Recursive character

Tries to split on paragraph, then sentence, then word boundaries, falling back only when a chunk is still too large. The most common default (e.g. LangChain's splitter).

03

Sentence-based

Groups whole sentences up to a token budget. Keeps grammatical units intact — useful for QA-style retrieval.

04

Semantic chunking

Embeds sentences and splits where meaning shifts significantly, rather than at a fixed size. Costs more compute up front, but chunks stay topically coherent.

05

Structure-aware

Splits along a document's own structure — markdown headers, HTML tags, code functions — so a chunk maps to one logical section.

06

Overlap

A sliding overlap (e.g. 50–100 tokens) between consecutive chunks prevents an answer-bearing sentence from being orphaned at a chunk edge.

05 · Deep Dive

Step 2 — Embeddings

An embedding model turns a chunk of text into a dense vector of floating-point numbers — a point in high-dimensional space — positioned so that semantically similar text ends up nearby.

What it captures

Meaning, not just words. "car" and "automobile" land close together even with zero shared characters, because the model learned meaning from context during training.

Dimensionality

Common sizes: 384 (MiniLM), 768 (BERT-family), 1536 (OpenAI text-embedding-3-small), 3072 (text-embedding-3-large). Higher dimensions capture more nuance at higher storage/compute cost.

Same model, both sides

The query and the documents must be embedded with the same model — vectors from different embedding models aren't comparable to each other.

Common models

OpenAI text-embedding-3, Azure OpenAI ada-002, Cohere embed-v3, open-source bge / e5 / sentence-transformers for self-hosting.

Which one to choose?

Prototyping or a small internal tool with no infra team → Chroma. Shipping a production SaaS feature and want to avoid running infrastructure → Pinecone. Data must stay on-prem or in your own cloud, or you need complex metadata filters and full control → Qdrant. In an enterprise Azure environment, Azure AI Search's vector support is also worth weighing against all three.

07 · Deep Dive

Step 4 — Search strategies

Retrieval isn't always pure vector search. Production systems usually blend approaches.

Keyword / lexical search

Matches exact terms using algorithms like BM25 or TF-IDF. Fast, explainable, and excellent at exact matches — product codes, error strings, names — but blind to synonyms or paraphrasing.

Semantic / vector search

Embeds the query and finds nearby vectors by cosine similarity, dot product, or Euclidean distance, using an ANN index such as HNSW. Understands meaning and paraphrase, but can miss exact rare terms an embedding wasn't trained to weight heavily.

Hybrid search

Runs keyword and semantic search in parallel, then merges the two ranked lists — commonly with Reciprocal Rank Fusion (RRF). Gets the recall of semantic search plus the precision of exact keyword matches. The default choice for most production RAG systems.

Metadata filtering

Narrows the search space by structured fields — date > 2024, department = "finance", access_level = "public" — before or alongside similarity search. Essential for multi-tenant and access-controlled RAG.

08 · Deep Dive

Cosine similarity, visually

Cosine similarity measures the angle between two vectors, ignoring their length — a good match when what matters is direction (meaning), not magnitude. Drag the sliders to see it move.

cos θ = 0.866
similarity(A, B) = (A · B) / (‖A‖ × ‖B‖)
Range: −1 (opposite) to 1 (identical direction). Two chunks about the same topic sit at a small angle → a score close to 1.
A narrow angle — these two vectors point in nearly the same direction. In embedding space, that means the text they represent is semantically close.

Step 5 — Re-ranking

Vector search is fast because it compares independently-computed embeddings (a bi-encoder). A cross-encoder reranker instead reads the query and each candidate together, which is slower but far more accurate — so it's applied only to the top handful of candidates the first pass already found.

10 · Why & Where

Benefits of using RAG

01

Fewer hallucinations

Answers are grounded in retrieved evidence instead of the model's memorized parameters.

02

Always current

Update the index and the system's knowledge updates instantly — no retraining cycle.

03

Private & domain data

Ground answers in internal docs, tickets, or codebases the base model never saw.

04

Cheaper than fine-tuning

Indexing text is far less costly than training runs, and iterates in minutes, not days.

05

Traceable answers

Retrieved chunks double as citations, so a user can verify where an answer came from.

06

Smaller models go further

Good retrieval can let a smaller, cheaper LLM match a larger one on knowledge-heavy tasks.
